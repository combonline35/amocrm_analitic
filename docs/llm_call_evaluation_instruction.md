# Инструкция LLM-оценки звонков

Документ описывает актуальную механику оценки разговоров в сервисе `amocrm_service`.
Секреты и API-ключи здесь не приводятся.

## 1. Текущий режим

- Провайдер анализа: OpenRouter.
- Режим: `openrouter_raw`.
- Модель аккаунта: `openai/gpt-4o-mini`.
- Транскрипт отправляется в LLM без обезличивания.
- Источник успешного анализа в базе: `openrouter_raw_qa_v1`.
- Если включен режим обезличивания, источник был бы `openrouter_anonymized_qa_v1`.
- Если OpenRouter не сработал, сохраняется `openrouter_error_fallback` с ошибкой и нулевой оценкой.
- Старые локальные анализы `local_qa_schema_v1`, `rule_based_v1` и `openrouter_error_fallback` считаются устаревшими и могут попадать на пересчет через LLM.

## 2. Какие звонки доходят до анализа

Автообработка работает по шагам:

1. Импортирует звонки из amoCRM/TELPHIN.
2. Берет только звонки, подходящие под фильтры аккаунта:
   - воронка: `6486458`;
   - этапы: `55323254`, `55323258`, `55330934`;
   - ответственные: `12070754`, `5940766`;
   - минимальная длительность: 30 секунд;
   - только новые после `started_at`.
3. Если у звонка есть ссылка на запись, проверяет доступность аудио.
4. Скачивает аудио.
5. Отправляет аудио на транскрибацию.
6. Если транскрипт сохранен, отправляет транскрипт в LLM-анализ.

Если у звонка `duration = 0`, `link = ""`, `call_result = "Пропущенный звонок"`, то аудио нет и расшифровывать нечего.

## 3. Запрос в OpenRouter

URL:

```text
https://openrouter.ai/api/v1/chat/completions
```

Параметры:

```json
{
  "model": "openai/gpt-4o-mini",
  "temperature": 0.1,
  "response_format": {"type": "json_object"}
}
```

System message:

```text
You are a strict sales QA evaluator. Return only a valid JSON object. Do not include markdown. The transcript can be raw or anonymized; do not infer hidden personal data.
```

User message состоит из публичного контекста и полного промпта:

```json
{
  "direction": "incoming/outgoing/unknown",
  "duration_seconds": 0,
  "product": "натяжные потолки",
  "privacy": "raw transcript is sent to external LLM"
}
```

После контекста добавляется полный промпт и транскрибация:

```text
Транскрибация:
"""
...текст разговора...
"""
```

## 4. Основной промпт

```text
РОЛЬ
Ты старший РОП и специалист контроля качества продаж в компании по натяжным потолкам.

КОНТЕКСТ
Продукт: натяжные потолки.
Главная цель первичного звонка: понять задачу клиента, выявить параметры объекта и записать клиента на бесплатный замер.
Оценивай только по фактам из транскрибации. Если данных нет, пиши "не определено" или 0.

ЧТО НУЖНО ОЦЕНИТЬ
1. Установление контакта: приветствие, тон, работа с именем, понятное представление.
2. Выявление потребностей: площадь, количество потолков/комнат, адрес/район, сроки, тип помещения, бюджет, кто принимает решение.
3. Презентация и ценность: объяснил ли менеджер пользу бесплатного замера, что будет после замера, почему клиенту удобно продолжить.
4. Запись на замер: предложил ли конкретные слоты, подтвердил ли дату/время/контакты, зафиксировал ли следующий шаг.
5. Работа с возражениями: цена, сроки, "подумаю", конкурент, нет ЛПР, не готов назвать параметры.
6. Управление разговором: кто вел инициативу, были ли паузы/потери, довел ли менеджер разговор до результата.

РЕКОМЕНДАЦИИ
Рекомендации менеджеру должны быть прикладными: что сказать, что уточнить, какое возражение закрыть, какой следующий шаг сделать.
Рекомендации РОП должны быть управленческими: что проверить в CRM, что разобрать с менеджером, какой риск по сделке, какой контроль поставить.

"Менеджер: ..."
"Клиент: ..."
Если роль неочевидна, определи её по смыслу фразы. Фразы про компанию, заявку, замер, площадь, слоты и уточняющие вопросы обычно говорит менеджер. Фразы согласия, сомнений, "не знаю", "хочу узнать", "мне нужно" обычно говорит клиент.

Обязательные ключи:

  "Объяснение оценки (вероятность продажи)": "",
  "Как продать?": "",
  "Рекомендации менеджеру": [""],
  "Рекомендации РОП": [""],
  "ЛПР?": "да/нет/не определено — с обоснованием",
  "Факты": [""],
  "Потребности": [""],
  "Боли": [""],
  "Возражения": [""],
  "Предложение соответствует потребностям": "да/частично/нет — почему",
  "Потребности закрыли": [""],
  "Потребности НЕ закрыли": [""],
  "Установление контакта (%)": "__%",
  "Объяснение (Установление контакта)": "",
  "Выявление потребностей (%)": "__%",
  "Объяснение (Выявление потребностей)": "",
```

## 5. Чек-лист AI

Главный итоговый балл сейчас считается по чек-листу AI, а не по старым 95 баллам.

```text
ЧЕК-ЛИСТ AI ДЛЯ ОЦЕНКИ ЗВОНКА
Оцени разговор дополнительно по чек-листу ниже. Ставь частичные баллы, если критерий выполнен частично. Итог чек-листа имеет сырой максимум 147, но итоговая оценка для UI нормализуется до 100.

Ветвление:
- Правила 5 и 9 взаимоисключающие.
- Если клиент в рамках одного звонка несколько раз просит цену/расчет ("сколько стоит", "рассчитайте", "так сколько получится", "сделайте расчет"), оценивай правило 5.
- Если клиент не спрашивал цену/расчет и не записался на замер, оценивай правило 9.
- Если клиент спрашивал цену/расчет, правило 9 ставь 0 и объясни "не применялось, сработала ветка правила 5".
- Если клиент не спрашивал цену/расчет, правило 5 ставь 0 и объясни "не применялось, сработала ветка правила 9 или запись на замер".

Правило закрытых вопросов:
- Если менеджер предложил замер через хороший "выбор без выбора" с двумя вариантами даты/времени, давай до 100% балла пункта.
- Если попытка была, но вопрос закрытый формата "удобно?", "сможете?", "записать?", на который можно ответить да/нет, давай примерно 30% максимума пункта.
- Если попытки нет, ставь 0.

Паузы:
- Если нет таймкодов, не измеряй реальные паузы. Оценивай "менеджер зависает" только по тексту: менеджер ответил и не задал встречный вопрос, потерял инициативу, не продвинул к замеру.

Правило 5 разбей внутри объяснения на подпункты: профиль 80/110/70/112, периметр, люстра/закладная/провод, светильники/вклейка/диаметр, гардина, полотно/микроны/оттенок, минимум 4 подхода с предложением замера. Не обязательно должны быть все параметры; важно минимум 4 разных подхода или похожих уточняющих параметра.

Правило 6:
- Сравни слова и словосочетания клиента со словами менеджера. Считай от всех смысловых слов клиента. Если менеджер явно перефразирует/использует около 30% клиентских слов/смыслов, ставь высокий балл.

Верни в JSON дополнительные ключи:
"Чек-лист AI": [
  {"id":1,"Критерий":"...","Балл":0,"Максимум":2,"Тег ошибки":"имя клиента","Статус":"выполнено/частично/не выполнено/не применялось","Доказательство":"короткая цитата или не определено","Комментарий":"почему такой балл"}
],
"ИТОГО по чек-листу (из 147)": 0,
"ИТОГО нормализовано (из 100)": 0,
"Критичные ошибки": ["теги ошибок, которые реально проявились"]
```

Критерии чек-листа:

| № | Макс. | Критерий | Тег ошибки |
|---|---:|---|---|
| 1 | 2 | Спросили имя и повторили 2 раза во время диалога | имя клиента |
| 2 | 5 | Все вопросы о замере: предложили 2 даты замера | закрытый вопрос |
| 3 | 7 | Первое предложение о замере при первом вопросе о цене: от 500 руб/м2, короткое пояснение про замерщика, дата замера выбором без выбора | душный разговор |
| 4 | 13 | После вопроса клиента менеджер отвечает кратко и сразу задает встречный вопрос | менеджер зависает |
| 5 | 47 | При повторных просьбах клиента о расчете менеджер задает параметры и минимум 4 раза предлагает замер в разных вариациях | нет много попыток |
| 6 | 8 | Менеджер использует не менее 30% слов/словосочетаний клиента | нет присоединения |
| 7 | 5 | Не говорить "мы не считаем по телефону" или "у нас такая система" | не согласны с клиентом |
| 8 | 3 | Есть вопрос "что смущает/останавливает от вызова замерщика?" | рано сдались |
| 9 | 47 | Если не записали на замер и клиент не спрашивал цену, минимум 4 вопроса о замере в формате выбор без выбора | мало попыток |
| 10 | 10 | Менеджер задал больше вопросов чем клиент, инициатива у менеджера | потеря инициативы |

## 6. Дополнительные показатели отчета

LLM также возвращает ключ `"Показатели отчета"`:

```json
{
  "Исход звонка": {
    "Запись на замер": "да/нет/не применимо",
    "Причина отказа": "конкретная причина",
    "Следующий шаг": "конкретное действие",
    "Вероятность продажи": "__%",
    "Объяснение вероятности": "почему такая вероятность"
  },
  "Профиль клиента": {
    "ЛПР": "да/нет/неясно + обоснование",
    "Факты": [],
    "Потребности": [],
    "Боли": [],
    "Возражения": []
  },
  "Попадание в потребности": {
    "Предложение соответствует потребностям": "да/частично/нет",
    "Потребности закрыли": [],
    "Потребности НЕ закрыли": []
  },
  "Навыки менеджера": {
    "Установление контакта": {"percent": 0, "explanation": ""},
    "Выявление потребностей": {"percent": 0, "explanation": ""},
    "Усиление боли": {"percent": 0, "explanation": ""},
    "Презентация": {"percent": 0, "explanation": ""},
    "Отработка возражений": {"percent": 0, "explanation": ""}
  },
  "Лидерство в разговоре": {
    "Кто лидер": "менеджер/клиент/на равных/неясно",
    "Почему": ""
  }
}
```

Правила:

- `"Запись на замер"` = `да`, только если реально согласовали замер, дату, окно или следующий контакт замерщика.
- Если это контроль после замера, ставится `не применимо`.
- `"Причина отказа"` должна быть конкретной: цена, хочет расчет без замера, думает, нет времени, выбрал конкурента, не ЛПР, район не обслуживается, контроль после замера и т.д.
- `"Предложение соответствует потребностям"` показывает, связал ли менеджер предложение с тем, что сказал клиент.
- Проценты навыков ставятся от 0 до 100.
- `"Кто лидер"` важен для РОПа: менеджер лидер, если он задает вопросы, ведет к замеру/следующему шагу и не просто отвечает клиенту.

## 7. Как считается итог

Если LLM вернул чек-лист:

```text
сырой балл = сумма "Балл" по 10 пунктам
максимум = 147
итог UI = round(сырой балл / 147 * 100)
```

Пример: `53 / 147 = 36 / 100`.

В UI показывается:

- `Чек-лист AI`: сырой балл из 147;
- `Итог 0-100`: нормализованный балл;
- `Критичные ошибки`: количество тегов ошибок;
- проценты навыков;
- рекомендации менеджеру;
- рекомендации РОПу;
- факты, потребности, боли, возражения;
- диалог с ролями.

Если чек-лист не пришел, сервис делает второй запрос в LLM только за чек-листом.

Если чек-листа все равно нет, используется старый fallback по 95 баллам.

## 8. Тональность

Тональность в базе считается от вероятности продажи:

```text
>= 70%: positive
< 40%: negative
иначе: neutral
```

В UI это переводится:

- `positive` -> `Позитивный`;
- `neutral` -> `Нейтральный`;
- `negative` -> `Негативный`;
- `error` -> `Ошибка анализа`.

## 9. Что сохраняется в базе

В `conversation_analysis.analysis_json` сохраняется:

- `source`;
- `model`;
- полный `qa_json` от LLM;
- используемый `analysis_prompt`;
- данные записи: сделка, контакт, направление, тип, ссылка на запись, время звонка.

В `conversation_analysis.metrics_json` сохраняется:

- `score_max`;
- `probability_percent`;
- `duration_seconds`;
- `score_blocks`;
- `checklist`;
- `report_metrics`;
- `percent_blocks`;
- `transcript_chars`.
